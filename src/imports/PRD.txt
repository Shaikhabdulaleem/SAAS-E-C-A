## 3.3 Multi-Tenant Enforcement Rules

```
RULE 1: organization_id filter
Every database query MUST include WHERE organization_id = ?
This is enforced at the service layer, not just controller

RULE 2: JWT token carries org context
Every authenticated request includes:
{ user_id, organization_id, role }
These values are injected from the token, never from request body

RULE 3: Record ownership
Every record has:
→ organization_id (tenant isolation)
→ created_by (user who created)
→ assigned_to (user responsible)

RULE 4: Role middleware
Each route declares required minimum role.
Middleware validates token role before route handler executes.
```

### Scheduling & Sending

```
SENDING FLOW:
1. User clicks Send or Schedule
2. System validates:
   → From email is verified
   → Audience has at least 1 subscribed contact
   → HTML content is not empty
   → Unsubscribe link is present
3. If scheduled: save scheduled_at, status=scheduled
4. If send now: status=sending, enqueue job
5. Background worker processes send queue:
   → Fetches contacts in batches of 500
   → Resolves template variables per contact
   → Calls SendGrid API (or Amazon SES)
   → Records email_sends row per recipient
   → Updates campaign stats on completion

SENDING INFRASTRUCTURE:
→ Email provider: SendGrid (primary)
   Fallback: Amazon SES
→ Queue: Bull Queue (Redis-backed)
→ Rate limiting: respect provider limits
→ Batch size: 500 per job cycle
→ Retry: 3 attempts on transient failures

SENDER VERIFICATION:
→ From email domain must be verified before use
→ Platform provides DKIM/SPF/DMARC setup guide
   (same flow as documented in EM.txt)
→ Verification check via DNS lookup
→ Unverified sender blocked from sending
```

### Tracking (Open / Click)

```
OPEN TRACKING:
→ 1×1 pixel tracking image injected into HTML
→ URL: /track/o/{tracking_token}
→ When pixel loads: write to email_events table
→ event_type = opened
→ Deduplicate: first open counts, subsequent
  opens increment open_count but one event row

CLICK TRACKING:
→ All links in email HTML wrapped at send time
→ URL: /track/c/{tracking_token}/{link_hash}
→ On click: write event, then 302 redirect to original URL
→ event_type = clicked
→ Original URL stored in email_links table

TRACKING TOKEN:
→ Unique per email_send row (per recipient per campaign)
→ JWT or HMAC-signed opaque token
→ Contains: email_send_id, organization_id
→ No PII in tracking URL itself

BOUNCE HANDLING:
→ SendGrid webhook → /webhooks/sendgrid
→ Events received: bounce, spam_report, unsubscribe
→ bounce hard → mark email in contacts as invalid
→ spam_report → add to global_unsubscribes
→ unsubscribe → add to global_unsubscribes
→ Webhook endpoint secured with signature verification

EMAIL_EVENTS TABLE:
→ id, organization_id
→ email_send_id, campaign_id, contact_id
→ event_type: [sent, opened, clicked,
               bounced, unsubscribed, spam]
→ metadata (JSONB: link clicked, user agent, etc.)
→ occurred_at
```

### CRM Chat Assistant

```
DEFINITION:
A conversational interface where users can
ask natural language questions about their
CRM data and get instant answers.

CAPABILITIES:
─────────────────────────────────────────
Data Retrieval:
→ "How many open deals do I have?"
→ "Show me all contacts at Acme Corp"
→ "Which deals are closing this month?"
→ "What's my pipeline value?"
→ "Who hasn't been contacted in 30 days?"

Summarization:
→ "Summarize my activity this week"
→ "Give me an overview of the Acme Corp account"
→ "What happened on the Johnson deal?"

Task Generation:
→ "Create a task to follow up with John tomorrow"
→ "Add a note to the Acme deal"

Action Execution (MVP subset):
→ Creating tasks (confirmed by user before execution)
→ Logging a note
→ Updating deal stage (with confirmation)

ARCHITECTURE:
─────────────────────────────────────────
Step 1: User sends message
Step 2: Backend receives message
Step 3: Build context prompt:
   → System instructions (role, capabilities, restrictions)
   → Organization context (user name, org name, date)
   → CRM data snapshot (relevant records fetched
     based on message intent classification)
Step 4: Send to OpenAI GPT-4o
Step 5: Parse response
Step 6: If action detected → execute with confirmation
Step 7: Return response to client

CONTEXT INJECTION STRATEGY:
→ Do NOT send entire database to OpenAI
→ Use intent classification first (fast regex or
  lightweight call) to determine what data to fetch
→ Fetch only relevant data (max 20 records)
→ Format as structured text in system prompt
→ Example context injection:

"CURRENT PIPELINE DATA:
 Open Deals: 12 | Total Value: $145,000
 Deals closing this month: 3
 [Deal 1] Acme Corp - $45,000 - Proposal stage
 [Deal 2] Beta Inc - $30,000 - Negotiation stage
 ..."

LIMITS:
→ Max 20 messages per session stored in context
→ Session resets after 2 hours of inactivity
→ Rate limit: 50 AI requests per user per day (MVP)
→ All AI sessions logged for debugging
```

 Always include one clear CTA.

 subject_lines[], preview_text, html_body,
 plain_text, cta_text"

User:
"Write a [type] email. Purpose: [description].
 Tone: [tone]. Length: [length].
 Context: [injected CRM data]"
```

### Deal Insights

```
DEFINITION:
AI analyzes individual deals and surfaces
actionable intelligence.

INSIGHTS GENERATED PER DEAL:
─────────────────────────────────────────
1. Deal Health Score (0-100)
   → Based on: days in current stage,
     activity recency, contact engagement,
     days until expected close
   → Color coded: Green (70+), Yellow (40-69), Red (<40)

2. Risk Flags (rule-based, not AI):
   → "No activity in 14+ days"
   → "Expected close date passed"
   → "No contact linked to deal"
   → "No activity since stage change"

3. Next Best Action (AI-generated):
   → Analyzes deal data and suggests:
     "Send a follow-up email to [contact] —
      last contact was 12 days ago and the
      proposal was sent without a response"

4. Similar Won Deals (rule-based):
   → Shows 3 past won deals with similar value/industry
   → "Deals like this typically close in 18 days"

WHEN INSIGHTS ARE CALCULATED:
→ On deal creation (async)
→ On stage change (async)
→ Every 24 hours via background job
→ On-demand when user opens deal detail
→ Stored in deal_insights table, not recalculated
  on every page load
```

### Daily Sales Summary

```
DEFINITION:
Automated AI-generated summary delivered
to each user every morning showing their
pipeline state and priorities.

DELIVERY:
→ Shown on dashboard on first login of the day
→ Email delivery (optional, user preference)
→ Generated at 7 AM in user's timezone

CONTENT:
─────────────────────────────────────────
1. Pipeline Snapshot
   → Total open deals + value
   → Deals won/lost yesterday
   → Stage movement yesterday

2. Today's Priorities (AI-ranked)
   → Up to 5 tasks due today
   → Up to 3 deals needing attention
   → Contacts not followed up in 7+ days

3. Email Campaign Update
   → Any campaigns sent yesterday
   → Open/click summary

4. AI Commentary (2-3 sentences)
   → Plain English summary
   → Generated by GPT-4o
   → Uses the structured data above as context

EXAMPLE OUTPUT:
"You have 14 open deals worth $182,000.
 3 deals are at risk — no activity in 14+ days.
 Your highest priority today: follow up with
 Sarah at Acme Corp whose proposal expires Friday.
 Yesterday's email campaign to 450 contacts
 achieved a 34% open rate."

GENERATION PROCESS:
→ Background job runs at 7 AM per user timezone
→ Fetches structured data via internal service calls
→ Builds context prompt
→ Calls GPT-4o (single API call per user)
→ Stores result in daily_summaries table
→ Served from cache on dashboard load
→ Not regenerated until next day
```

## 6.4 AI Layer Data Flow

```
REQUEST FLOW (CRM Chat):
─────────────────────────────────────────────

1. User types message in AI chat UI
   POST /api/v1/ai/chat
   { session_id, message }

2. AI Service receives request

3. Intent classification (fast, synchronous):
   → Simple keyword/pattern matching
   → Classifies as: data_query | action | general
   → Determines what CRM data to fetch
   → Takes < 50ms, no API call needed

4. Context builder fetches relevant data:
   → If intent = pipeline_query:
     Fetch deals summary for user's org
   → If intent = contact_query:
     Fetch matching contacts (max 20)
   → If intent = deal_specific:
     Fetch specific deal + activities
   → If intent = general:
     Fetch lightweight org summary

5. Build OpenAI messages array:
   [
     { role: "system", content: SYSTEM_PROMPT
       + org_context + crm_data },
     ...last 10 messages from session,
     { role: "user", content: user_message }
   ]

6. Call OpenAI GPT-4o
   model: gpt-4o
   max_tokens: 1000
   temperature: 0.3 (factual, consistent)

7. Parse response:
   → Check for action markers in response
   → If action detected: extract parameters
   → Present to user for confirmation before executing

8. Save messages to ai_messages table

9. Return response to client with:
   { message, actions[], session_id }

10. If action confirmed by user:
    → Execute (create task, update deal, etc.)
    → Log as system activity

TOTAL LATENCY TARGET: < 3 seconds end-to-end
```

## 6.5 Email Sending Data Flow

```
CAMPAIGN SEND FLOW:
─────────────────────────────────────────────

1. User clicks "Send Campaign"
   POST /api/v1/campaigns/:id/send

2. Backend validates:
   → Campaign status = draft
   → from_email domain is verified
   → contact_list has subscribers
   → HTML content not empty
   → Unsubscribe placeholder exists

3. Set campaign status = sending
   Set total_recipients = count of list members
   minus global_unsubscribes

4. Enqueue Bull job: campaign_send:{campaign_id}

5. Background Worker picks up job:
   a. Fetch list members in batches (500)
   b. For each batch:
      → Check not in global_unsubscribes
      → Check contact.unsubscribed = false
      → Check contact.email_invalid = false
      → Generate tracking_token (HMAC)
      → Replace template variables
      → Wrap all links with tracking URLs
      → Inject tracking pixel
      → Call SendGrid API (batch send)
      → Insert email_sends rows (status=sent)
      → Update campaign stats_sent counter

6. On worker completion:
   → Set campaign status = sent
   → Set campaign sent_at = NOW()

7. SendGrid sends emails to recipients

8. SendGrid webhooks fire to /webhooks/sendgrid:
   → open event → update email_sends, create event
   → click event → update email_sends, create event
   → bounce → update email_sends, flag contact
   → unsubscribe → add to global_unsubscribes

9. Campaign stats update on each webhook event
```

# SECTION 11: KEY PRODUCT RISKS

---

## Technical Risks

```
RISK 1: OpenAI Latency and Cost
Severity: HIGH
Description:
  AI chat response times can be 3-8 seconds.
  At scale, OpenAI API costs can spike unpredictably.
  Token limits may truncate large CRM context payloads.

Mitigation:
  → Implement smart context trimming (max 3,000 tokens
    for CRM data, prioritize most relevant records)
  → Cache daily summaries (generate once, serve all day)
  → Rate limit AI requests per user (50/day on MVP)
  → Use GPT-4o-mini for lightweight queries (classification,
    subject line generation) and GPT-4o only for chat
  → Stream responses to improve perceived latency
  → Set monthly OpenAI spend alert at $200

─────────────────────────────────────────────────────

RISK 2: Email Deliverability
Severity: HIGH
Description:
  Shared sending IP reputation can affect all clients.
  If one customer sends spam, everyone suffers.
  Domain authentication complexity causes setup failure.

Mitigation:
  → Enforce sender domain verification before any send
  → Monitor spam complaint rates per organization
  → Suspend sending if complaint rate > 0.15%
  → Enforce unsubscribe link in every campaign
  → Use SendGrid dedicated IPs once volume justifies ($50/mo)
  → Build domain setup wizard with live DNS verification

─────────────────────────────────────────────────────

RISK 3: Multi-Tenant Data Isolation
Severity: CRITICAL
Description:
  A bug in tenant isolation logic could expose
  one organization's data to another.
  This would be catastrophic for trust and legally.

Mitigation:
  → organization_id added at service layer,
    never trusted from request body
  → Write automated tests that attempt
    cross-tenant data access (must fail)
  → All queries use parameterized statements only
    (no raw string concatenation)
  → Row-level security policies in PostgreSQL
    as backup layer (defense in depth)
  → Security audit before production launch

─────────────────────────────────────────────────────

RISK 4: Email Queue Reliability
Severity: MEDIUM
Description:
  If Bull/Redis fails during a campaign send,
  some recipients may get duplicate emails or none.

Mitigation:
  → Idempotency key on each email_send row
  → Check status before re-processing (skip if sent)
  → Dead letter queue for failed jobs (manual retry)
  → SendGrid message ID stored on email_send row
    to detect duplicate API calls
  → Maximum 3 retries with exponential backoff

─────────────────────────────────────────────────────

RISK 5: Database Performance at Scale
Severity: MEDIUM
Description:
  Activity timeline queries with large activity
  tables could slow to unacceptable speeds.
  Full-text search on contacts needs to be fast.

Mitigation:
  → All FK columns have indexes (included in schema)
  → Activities query uses (organization_id, logged_at DESC)
    composite index
  → Pagination enforced on all list endpoints (max 100/page)
  → Full-text search on contacts uses PostgreSQL
    GIN index on (first_name, last_name, email) tsvector
  → Query performance monitoring from day 1 (pg_stat_statements)
  → Connection pooling via PgBouncer or Neon connection pooling
```

---

## Product-Market Risks

```
RISK 6: Feature Parity Gap vs. Free Tools
Severity: MEDIUM
Description:
  HubSpot has a free CRM. Mailchimp has a free email tier.
  Users may use free tools separately rather than pay for combined.

Mitigation:
  → Compete on AI integration depth, not feature count
  → Time-to-value is the differentiator: everything in one place
    configured in under 60 minutes
  → Price below HubSpot starter ($45/mo) for comparable features
  → Don't try to match every HubSpot feature, be better at
    the 20% of features 80% of users actually use

─────────────────────────────────────────────────────

RISK 7: AI Over-Promising / Under-Delivering
Severity: MEDIUM
Description:
  Users with high AI expectations may be disappointed
  if the assistant gives wrong answers about their data
  or takes wrong actions.

Mitigation:
  → Always require confirmation before AI takes any action
  → Show clearly what data sources AI used for its answer
  → Gracefully decline when confidence is low:
    "I don't have enough data to answer that confidently"
  → Do not make AI seem like it can do things it can't
  → Start with clearly bounded capabilities, expand gradually

─────────────────────────────────────────────────────

RISK 8: Onboarding Drop-off
Severity: MEDIUM
Description:
  CRM setup requires effort upfront (importing data,
  setting up pipelines, configuring email domains).
  Users who don't get immediate value will churn.

Mitigation:
  → Sample data mode: offer to pre-populate with
    demo contacts/deals so new users see a working product
  → Onboarding checklist drives first 5 actions
  → AI assistant greets new users and guides them:
    "I see you haven't imported any contacts yet.
     Want help doing that?"
  → Email drip during trial: day 1, day 3, day 7 activation tips
  → Time-to-first-value < 15 minutes is a hard product goal
```

## 3.3 Multi-Tenant Enforcement Rules

```
RULE 1: organization_id filter
Every database query MUST include WHERE organization_id = ?
This is enforced at the service layer, not just controller

RULE 2: JWT token carries org context
Every authenticated request includes:
{ user_id, organization_id, role }
These values are injected from the token, never from request body

RULE 3: Record ownership
Every record has:
→ organization_id (tenant isolation)
→ created_by (user who created)
→ assigned_to (user responsible)

RULE 4: Role middleware
Each route declares required minimum role.
Middleware validates token role before route handler executes.
```

## 14.1 Authentication Security

```
ACCESS TOKEN (JWT):
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Algorithm:        RS256 (asymmetric — private key signs,
                  public key verifies)
                  NOT HS256 (symmetric keys are riskier
                  in microservice architectures)
Expiry:           15 minutes
Storage:          JavaScript memory only
                  (NOT localStorage, NOT sessionStorage)
Transport:        Authorization: Bearer <token> header
Payload:          {
                    sub: user_id,
                    org: organization_id,
                    role: user_role,
                    iat: issued_at,
                    exp: expiry,
                    jti: unique_token_id
                  }
Key rotation:     Private key rotatable without service restart
                  Public keys served at /.well-known/jwks.json

REFRESH TOKEN:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Format:           Opaque cryptographically random string (32 bytes)
                  Generated via: crypto.randomBytes(32).toString('hex')
Expiry:           7 days (rolling)
Storage:          httpOnly, Secure, SameSite=Strict cookie
                  Domain scoped to app domain only
DB record:        refresh_tokens table
                  { id, user_id, token_hash, expires_at,
                    revoked_at, created_at, ip_address,
                    user_agent }
                  Token stored as SHA-256 hash (never plaintext)
Rotation:         New refresh token issued on every use
                  Old token immediately revoked (rotation prevents reuse)
Revocation:       Token revoked on logout
                  All user tokens revoked on password change
                  All user tokens revoked on role change
                  All org tokens revocable by admin (emergency)

REFRESH TOKENS TABLE:
```sql
CREATE TABLE refresh_tokens (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id         UUID NOT NULL REFERENCES users(id)
                  ON DELETE CASCADE,
  organization_id UUID NOT NULL,
  token_hash      VARCHAR(64) NOT NULL UNIQUE,
                  -- SHA-256 of the actual token
  expires_at      TIMESTAMPTZ NOT NULL,
  revoked_at      TIMESTAMPTZ,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  ip_address      INET,
  user_agent      TEXT
);

CREATE INDEX idx_refresh_tokens_user
  ON refresh_tokens(user_id)
  WHERE revoked_at IS NULL;

CREATE INDEX idx_refresh_tokens_hash
  ON refresh_tokens(token_hash)
  WHERE revoked_at IS NULL;
```

SESSION SECURITY RULES:
→ Concurrent sessions allowed (multiple devices)
→ Maximum 10 active sessions per user
  (11th session revokes oldest)
→ Session list visible to user in /settings/profile
→ User can revoke individual sessions
→ Admin can revoke all sessions for a user
```

## 14.2 Password Security

```
HASHING STANDARD:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Algorithm:    bcrypt
Cost factor:  12 (balances security vs performance)
              Benchmark: ~250ms on 2 vCPU instance
              Acceptable for login flows
Library:      bcryptjs (Node.js) or native bcrypt
Storage:      password_hash column (VARCHAR 255)
              Only hash stored, never plaintext

PASSWORD POLICY (enforced at registration and change):
→ Minimum length: 8 characters
→ Maximum length: 128 characters (prevent DoS via huge inputs)
→ Must contain: 1 uppercase, 1 lowercase, 1 number
→ Common password list check: top 10,000 common passwords rejected
→ Cannot reuse last 5 passwords
   (requires password_history table — Phase 2)

PASSWORD RESET FLOW:
→ Token: crypto.randomBytes(32).toString('hex')
→ Token hash stored in DB (SHA-256), never plaintext
→ Expiry: 1 hour
→ Single-use: token invalidated after first successful use
→ Rate limited: max 3 reset requests per email per hour
→ Email contains no sensitive info beyond link
→ Success: all sessions revoked, user must re-login

TIMING ATTACK PREVENTION:
→ Login always takes same time whether user exists or not
→ Use bcrypt.compare() even for non-existent users
   (compare against dummy hash to normalize timing)
→ Password reset endpoint returns same message
   whether email exists or not:
   "If that email exists, you'll receive a reset link"
```

## 14.3 Role-Based Access Control Enforcement

```
ENFORCEMENT ARCHITECTURE:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Layer 1: Route Middleware (first line of defense)
─────────────────────────────────────────────────
Every route declares its access requirements:

router.delete('/contacts/:id',
  authenticate,              // verify JWT
  requireRole('manager'),    // check minimum role
  requireOrgMembership,      // verify user in org
  contactController.delete
);

requireRole middleware:
const ROLE_HIERARCHY = {
  owner:   5,
  admin:   4,
  manager: 3,
  member:  2,
  viewer:  1
};

function requireRole(minimumRole) {
  return (req, res, next) => {
    const userLevel = ROLE_HIERARCHY[req.user.role];
    const requiredLevel = ROLE_HIERARCHY[minimumRole];
    if (userLevel < requiredLevel) {
      return res.status(403).json({
        error: 'INSUFFICIENT_PERMISSIONS',
        message: 'You do not have permission to perform this action',
        required_role: minimumRole,
        your_role: req.user.role
      });
    }
    next();
  };
}

Layer 2: Service Layer (business rule enforcement)
─────────────────────────────────────────────────
For "OWN record" rules (member can only edit own records):

async function updateContact(contactId, updates, requestingUser) {
  const contact = await db.contacts.findOne({
    where: {
      id: contactId,
      organization_id: requestingUser.organization_id
    }
  });

  if (!contact) throw new NotFoundError();

  // Member can only edit their own assigned contacts
  if (requestingUser.role === 'member' &&
      contact.assigned_to !== requestingUser.id) {
    throw new ForbiddenError('You can only edit contacts assigned to you');
  }

  return await db.contacts.update(contactId, updates);
}

Layer 3: Database (last resort, safety net)
─────────────────────────────────────────────────
PostgreSQL Row Level Security (RLS) as defense-in-depth
(see Section 22 for full RLS implementation)

ROLE CHANGE RULES:
→ Role changes take effect on next API request
→ JWT refresh immediately on role change
   (invalidate old tokens via jti revocation list in Redis)
→ Downgrade: owner cannot downgrade themselves
→ Owner role: only transferable by current owner
   (POST /organization/transfer-ownership)
→ Admin cannot promote to owner
→ Audit log entry required for all role changes
```

## 14.4 Encryption Standards

```
ENCRYPTION IN TRANSIT:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Protocol:     TLS 1.2 minimum, TLS 1.3 preferred
Enforcement:  HSTS header with max-age=31536000
              includeSubDomains; preload
Certificate:  Managed by Vercel (frontend) and
              deployment platform (backend)
              Auto-renewed, alerts at 30 days before expiry
Cipher suite: Modern cipher suites only
              (configured at load balancer / platform level)
              No RC4, DES, or 3DES

API-to-API:   All internal service calls also over TLS
Database:     TLS connection to PostgreSQL (require SSL)
Redis:        TLS connection enforced (Upstash enforces by default)
SendGrid:     All API calls over TLS (SDK enforced)
OpenAI:       All API calls over TLS (SDK enforced)

HTTP SECURITY HEADERS (applied via helmet.js):
→ Content-Security-Policy: enforced, report-only initially
→ X-Content-Type-Options: nosniff
→ X-Frame-Options: DENY
→ X-XSS-Protection: 1; mode=block
→ Referrer-Policy: strict-origin-when-cross-origin
→ Permissions-Policy: camera=(), microphone=(), geolocation=()
→ Cross-Origin-Opener-Policy: same-origin

ENCRYPTION AT REST:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Database:     AES-256 encryption at storage level
              (Neon PostgreSQL encrypts all data at rest)
File storage: AES-256 (AWS S3 server-side encryption SSE-S3)
Redis:        Upstash encrypts at rest by default
Backups:      Encrypted with AES-256 before upload

SENSITIVE FIELD ENCRYPTION (application level):
The following fields are encrypted at the application
layer before storage, in addition to disk encryption:

Fields requiring application-level encryption:
→ sendgrid_api_key (in organizations.settings)
→ openai_api_key (if customer-provided, future feature)
→ smtp_credentials (future feature)
→ stripe_customer_id (stored but not sensitive — no extra enc.)

Encryption approach for sensitive fields:
→ AES-256-GCM
→ Encryption key stored in environment variable
→ Key derivation: PBKDF2 with application secret
→ Different initialization vector (IV) per encrypted value
→ IV stored alongside ciphertext (base64 encoded)

Example implementation:
const encrypt = (plaintext) => {
  const iv = crypto.randomBytes(16);
  const cipher = crypto.createCipheriv(
    'aes-256-gcm', ENCRYPTION_KEY, iv
  );
  const encrypted = Buffer.concat([
    cipher.update(plaintext, 'utf8'),
    cipher.final()
  ]);
  const authTag = cipher.getAuthTag();
  return Buffer.concat([iv, authTag, encrypted])
    .toString('base64');
};
```

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

→ Application logs (console, Sentry) must never contain:
  raw email addresses, names, phone numbers
→ Use IDs in logs: "Processing contact ID: abc-123"
  NOT "Processing contact: john@company.com"
→ Sentry beforeSend hook scrubs PII fields from error reports

Rule 2: No PII in URLs
→ Email addresses never appear in URL query parameters
→ IDs used for all resource references in URLs
→ Tracking tokens contain no PII (opaque tokens only)

→ When injecting CRM data into AI context:
  Use structured format: "Contact: John S. | Company: Acme"
  NOT raw database dump with all fields
→ AI session logs in DB are subject to same retention policy

Rule 4: Data minimization for AI
→ AI context includes only fields necessary for the query
→ Phone numbers and personal addresses not sent to OpenAI
  unless explicitly needed for the response

Rule 5: Deletion propagation
→ When a contact is deleted (hard delete):
  email_sends.contact_id SET NULL (email preserved for stats)
  email_events.contact_id SET NULL
  activities.contact_id SET NULL
  deal_contacts entries deleted
  contact_list_members entries deleted
→ email_sends.email field is anonymized:
  UPDATE email_sends SET email = 'deleted@redacted.invalid'
  WHERE contact_id = $deleted_id

Rule 6: Export controls
→ CSV exports of contacts log an audit event
→ Exports include only fields the requesting user can see
→ Bulk exports rate limited: max 3 exports per hour per user
→ Download links expire after 1 hour (pre-signed URLs from S3)

## 15.1 Rate Limiting Architecture

```
IMPLEMENTATION:
→ Library: express-rate-limit + rate-limit-redis
→ Storage: Redis (ensures limits work across multiple API instances)
→ Key strategy: composite key per route tier
→ Response on limit exceeded: HTTP 429
  { error: "RATE_LIMIT_EXCEEDED",
    message: "Too many requests",
    retry_after: <seconds>,
    limit: <requests>,
    window: <seconds> }
→ Headers returned on every request:
  X-RateLimit-Limit: 100
  X-RateLimit-Remaining: 47
  X-RateLimit-Reset: 1704067200
  Retry-After: 60 (only on 429)

## 15.2 Rate Limits Per Endpoint Tier

```
TIER A — Authentication (strictest)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Key:    IP address + email (for login)
        IP address only (for register, forgot-password)

  → 5 attempts per 15 minutes per IP+email combination
  → 20 attempts per 15 minutes per IP (total)
  → On 5th failure: account lockout for 15 minutes

  → After lockout: 429 returned without attempting auth
  → Lockout lifted automatically after 15 minutes

POST /auth/register:
  → 3 accounts per hour per IP address
  → 10 accounts per day per IP address

POST /auth/forgot-password:
  → 3 requests per hour per email address
  → 10 requests per hour per IP address

POST /auth/refresh:
  → 30 requests per 15 minutes per user_id
  → Protects against token refresh storms

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

General read endpoints (GET):
  → 1,000 requests per 15 minutes per organization
  → Burst: 200 requests per minute per organization

General write endpoints (POST/PUT/DELETE):
  → 300 requests per 15 minutes per organization
  → Burst: 60 requests per minute per organization

Bulk operations (import, bulk actions):
  → 10 requests per hour per organization
  → Applied to: POST /contacts/import,
                POST /lists/:id/members (bulk add)

TIER C — Email Operations (campaign sending)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Key:    organization_id

  → Growth plan:   30 requests per hour per user

POST /ai/generate-email:
  → Starter plan:  5 requests per hour per user
  → Growth plan:   20 requests per hour per user
  → Pro plan:      50 requests per hour per user

  → Starter: 100 AI requests/month per user
  → Growth:  500 AI requests/month per user
  → Pro:     2,000 AI requests/month per user

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

## 15.3 Brute Force Protection

```
LOGIN BRUTE FORCE:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Progressive lockout strategy:
  Attempts 1-3:   Normal response time + error message
  Attempts 4-5:   Add 2-second artificial delay + warning
                  "X attempts remaining before lockout"
  Attempt 6+:     Lockout for 15 minutes
                  "Account temporarily locked. Try again in 15 minutes."
                  Audit log: login_blocked_rate_limit (severity: warning)
  After 50 total attempts in 24 hours on same IP:
                  IP blocked for 24 hours
                  Alert sent to system admin

Redis keys used:
  login_attempts:{email}           → count, expires in 15 min
  login_lockout:{email}            → locked flag, expires in 15 min
  ip_login_attempts:{ip}           → count, expires in 24 hours
  ip_blocked:{ip}                  → blocked flag, expires in 24 hours

CREDENTIAL STUFFING DETECTION:
→ Alert triggered if:
  > 100 failed logins from same IP in 1 hour
  > 50 different email addresses attempted from same IP
  > Login attempts spike > 10x normal rate in 15-minute window
→ Alert goes to: system admin email + Slack webhook (if configured)

ENUMERATION PREVENTION:
→ POST /auth/login:
  Same HTTP status (401) and response time
  whether email exists or not
  Same message: "Invalid email or password"
→ POST /auth/forgot-password:
  Always returns 200 with same message
  "If that email exists, a reset link has been sent"
→ POST /users/invite:
  Returns 200 whether email already exists or not
  Actual behavior varies (send invite or notify existing user)
  but response is identical to prevent enumeration
```

## 15.4 Abuse Detection Rules

```
AUTOMATED ABUSE SIGNALS (trigger alerts):
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

SIGNAL 1: Spam complaint rate spike
  Threshold: > 0.5% spam complaints on campaign
  Action:    Suspend org email sending immediately
             Send email to org owner + internal alert
             Require admin review to unsuspend

SIGNAL 2: Bounce rate spike
  Threshold: > 10% hard bounce rate on campaign
  Action:    Pause campaign send (if in progress)
             Flag organization for review
             Recommend list cleaning

SIGNAL 3: Rapid contact import
  Threshold: > 50,000 contacts imported in 1 hour
  Action:    Queue import for manual review
             Notify internal team
             Auto-approve after 4 hours if no manual action

SIGNAL 4: Unusual API pattern
  Threshold: > 10,000 API requests in 5 minutes
             from single organization
  Action:    Temporary 30-minute rate limit reduction to 10%
             Alert sent internally

SIGNAL 5: AI prompt injection attempt
  Detection: System prompt injection patterns in AI input:
             "ignore previous instructions",
             "you are now", "pretend you are",
             "system:", "[SYSTEM]", etc.
  Action:    Request blocked with 400 error
             Audit log entry (severity: warning)
             Count tracked: > 5 attempts = flag org for review

SIGNAL 6: Cross-tenant access attempt
  Detection: organization_id in URL/body does not match JWT
  Action:    Request blocked with 403 error
             Audit log entry (severity: critical)
             Immediate alert to security team

IMPLEMENTATION:
→ Signals 1-4: Background job runs every 5 minutes
  Queries Redis counters and DB aggregates
  Posts to internal alert webhook (Slack + email)
→ Signals 5-6: Detected synchronously in middleware
  Immediately block and log

INTERNAL ALERTING CHANNELS:
→ Slack: #security-alerts webhook
→ Email: security@nexushq.com (internal)
→ PagerDuty: for critical signals only (Signal 6)
```

## 16.1 Queue System Overview

```
TECHNOLOGY: BullMQ (Redis-backed)
Redis requirement: Redis 6+ (Upstash Redis compatible)

QUEUE ARCHITECTURE:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

QUEUE NAME            PRIORITY    CONCURRENCY    PURPOSE
──────────────────────────────────────────────────────────
email:critical        1 (highest) 20             Single email
                                                 send ops
email:campaigns       2           5              Bulk campaign
                                                 sending
ai:insights           3           3              Deal insight
                                                 generation
ai:summaries          4           10             Daily summary
                                                 generation
analytics:tracking    5           20             Open/click event
                                                 processing
maintenance:cleanup   6 (lowest)  2              Data cleanup
                                                 and archival

QUEUE ISOLATION RATIONALE:
→ email:critical is separate from email:campaigns so that
  transactional single emails (invite, password reset) are
  never delayed by a large bulk campaign send
→ AI queues are separate from email queues so an OpenAI
  slowdown does not affect email delivery
→ analytics:tracking is high concurrency because tracking
  events are high volume but very fast to process
→ maintenance has lowest priority, never affects user-facing ops

WORKER PROCESS ARCHITECTURE:
→ Each queue has dedicated worker process(es)
→ Workers are separate Node.js processes from API server
→ Deployed as separate service: nexushq-workers
→ Scales independently from API server
→ Worker processes: 1 process per queue type (MVP)
  Scale to 2+ processes per queue as volume grows

BULL DASHBOARD:
→ bull-board or arena UI deployed at /admin/queues
→ Protected by admin role
→ Shows: queue depth, processing rate, failed jobs, retries
```

## 16.2 Job Definitions

```
JOB 1: campaign_send
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Queue:          email:campaigns
Trigger:        POST /campaigns/:id/send
                Scheduled campaigns (cron check every minute)

Payload:
{
  campaign_id: string,
  organization_id: string,
  batch_number: number,        // which batch of contacts
  batch_offset: number,        // pagination offset
  idempotency_key: string      // campaign_id + batch_number
}

Processing steps:
1. Verify campaign status = 'sending' (idempotency check)
2. Fetch contact batch (LIMIT 500 OFFSET batch_offset)
3. Filter: remove unsubscribed, invalid email, in global_unsubscribes
4. For each contact: generate tracking token, wrap links
5. Call SendGrid batch send API
6. INSERT email_sends rows (status=sent)
7. UPDATE campaign stats_sent += batch_size
8. If more contacts remain: enqueue next batch job
9. If final batch: UPDATE campaign status='sent', sent_at=NOW()

Idempotency:
→ Before any DB writes, check:
  SELECT COUNT(*) FROM email_sends
  WHERE campaign_id = ? AND batch_idempotency_key = ?
  If count > 0: skip processing, mark job complete
→ batch_idempotency_key stored on email_sends rows

On failure:
→ Retry (see 16.3)
→ If batch fails after max retries:
  Set campaign to 'failed' status
  Send email notification to campaign creator

JOB 2: deal_insight_generate
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Queue:      ai:insights
Trigger:    Deal created, Deal stage changed,
            Nightly batch job (all open deals)

Payload: { deal_id: string, organization_id: string,
           trigger: 'create' | 'stage_change' | 'nightly' }

Processing steps:
1. Fetch deal + activities (last 30 days) + contacts
2. Calculate health score (rule-based formula, no AI)
3. Generate risk flags (rule-based checks)
4. Call OpenAI for next_best_action text (if health < 60)
5. Upsert deal_insights record
6. If health changed significantly (>20 points): notify
   assigned user via in-app notification

Timeout: 15 seconds (OpenAI call max wait)

JOB 3: daily_summary_generate
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Queue:      ai:summaries
Trigger:    Cron: every day at 7:00 AM per user timezone
            (scheduler checks users grouped by timezone hourly)

Payload: { user_id: string, organization_id: string,
           summary_date: string }

Processing steps:
1. Check: daily_summaries record already exists for today? Skip.
2. Fetch structured data (pipeline, tasks, campaigns)
3. Build context prompt
4. Call OpenAI GPT-4o
5. INSERT daily_summaries record
6. If user has email notifications enabled: send summary email

Idempotency:
→ Check existing record before processing
→ INSERT ... ON CONFLICT (user_id, summary_date) DO NOTHING

JOB 4: email_event_process
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Queue:          analytics:tracking
Trigger:        Tracking pixel / click endpoint calls
                SendGrid webhook events

Payload: { event_type: string, tracking_token: string,
           metadata: object, occurred_at: string }

Processing steps:
1. Decode tracking token → get email_send_id, organization_id
2. Verify email_send exists and belongs to correct org
3. Insert email_events row
4. Update email_sends (open_count, click_count, first_opened_at)
5. Update email_campaigns stats counters (atomic increment)
6. If activity logging enabled: create contact activity entry

Volume: up to 100,000 events/hour during campaign sends
Concurrency: 20 workers for this queue

JOB 5: contact_import_process
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Queue:      email:campaigns (reused, lower priority)
            Actually: separate queue: data:imports
Trigger:    POST /contacts/import (after CSV upload to S3)

Payload: { import_id: string, organization_id: string,
           s3_key: string, user_id: string,
           field_mapping: object }

Processing steps:
1. Download CSV from S3
2. Parse CSV rows in streaming fashion (don't load all to memory)
3. Validate each row (required fields, email format)
4. Check duplicates (batch query against existing emails)
5. Batch INSERT valid contacts (500 per transaction)
6. Track progress in Redis: import_progress:{import_id}
7. Send completion notification to user (in-app + optional email)
8. Clean up S3 file after processing

JOB 6: dns_verification_check
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Queue:      maintenance:cleanup
Trigger:    Cron: every 6 hours
            Manual trigger: POST /domains/:id/verify

Payload: { sender_domain_id: string,
           organization_id: string }

Processing steps:
1. Fetch sender_domain record
2. DNS lookup: check SPF TXT record
3. DNS lookup: check DKIM CNAME/TXT records
4. DNS lookup: check DMARC TXT record
5. Update sender_domains verification columns
6. If newly verified: audit log + notify org owner
7. If was verified but now failing: alert org owner

JOB 7: data_cleanup
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Queue:      maintenance:cleanup
Trigger:    Cron: daily at 3:00 AM UTC

Processing steps:
1. Delete expired refresh_tokens
2. Delete expired email_verification_tokens
3. Delete expired password_reset_tokens
4. Archive email_events older than retention policy
5. Delete soft-deleted records past retention window
6. Aggregate old campaign stats (see Section 20)
7. Update contact.last_activity_at from activities table
```

## 16.3 Retry Strategy

```
RETRY CONFIGURATION PER QUEUE:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

email:critical:
  attempts:     5
  backoff:      exponential, initial delay 1 second
                attempt 1: immediate
                attempt 2: 1s
                attempt 3: 4s
                attempt 4: 16s
                attempt 5: 64s
  remove_on_complete: true
  remove_on_fail:     false (kept in dead letter queue)

email:campaigns:
  attempts:     3
  backoff:      exponential, initial delay 5 seconds
                attempt 1: immediate
                attempt 2: 5s
                attempt 3: 25s
  remove_on_complete: true
  remove_on_fail:     false

ai:insights:
  attempts:     3
  backoff:      exponential, initial delay 10 seconds
  timeout:      20000ms (20 seconds per attempt)
                If OpenAI takes >20s, job times out and retries
  remove_on_complete: true
  remove_on_fail:     false

ai:summaries:
  attempts:     2
  backoff:      fixed, 30 seconds between attempts
  timeout:      30000ms
  remove_on_complete: true
  remove_on_fail:     true (not critical, skip if failing)

analytics:tracking:
  attempts:     3
  backoff:      exponential, initial delay 500ms
  remove_on_complete: true
  remove_on_fail:     false

maintenance:cleanup:
  attempts:     2
  backoff:      fixed, 60 seconds
  remove_on_complete: true
  remove_on_fail:     false

DEAD LETTER QUEUE HANDLING:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Failed jobs (exhausted all retries) are:
1. Moved to failed job list in BullMQ (automatic)
2. Counted in metrics (alert if > 10 failures/hour)
3. Viewable in Bull Dashboard by admin
4. For campaign send failures:
   → Campaign creator notified via email
   → Campaign status set to 'failed'
   → Admin can retry via dashboard manually
5. For AI failures:
   → Silent fail (non-critical)
   → Logged for monitoring

RETRY DECISION LOGIC:
→ Retry on:
  Network errors (ECONNREFUSED, ETIMEDOUT)
  Rate limit errors (429 from SendGrid, OpenAI)
  Transient DB errors (connection pool exhausted)
  5xx errors from external APIs
→ Do NOT retry on:
  Invalid data errors (bad email format, missing fields)
  Authentication errors (invalid API keys)
  Business rule violations (campaign already sent)
  4xx errors from external APIs (except 429)
```

## 16.4 Job Prioritization & Scheduling

```
SCHEDULED JOBS (CRON):
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

JOB                     SCHEDULE        DESCRIPTION
──────────────────────────────────────────────────────────
campaign_scheduler      every 1 min     Check scheduled
                                        campaigns, enqueue sends
daily_summary_batch     every 1 hour    Find users whose 7AM
                                        timezone window just
                                        passed, enqueue summaries
deal_insight_nightly    daily 2AM UTC   Refresh all open deal insights
dns_verification        every 6 hours   Check all sender domains
data_cleanup            daily 3AM UTC   Cleanup expired data
email_quota_reset       1st of month    Reset monthly email counters
                        1AM UTC         for all organizations

SCHEDULER IMPLEMENTATION:
→ Bull repeatable jobs for all cron schedules
→ Cron expressions stored in code, not DB
→ Scheduler runs in a single dedicated process
  (not scaled horizontally to prevent duplicate executions)
→ Lock mechanism: Redis distributed lock prevents duplicate
  scheduler runs if multiple scheduler instances exist

CAMPAIGN SCHEDULER DETAIL:
→ Runs every minute
→ Queries:
  SELECT id FROM email_campaigns
  WHERE status = 'scheduled'
    AND scheduled_at <= NOW()
    AND scheduled_at > NOW() - INTERVAL '5 minutes'
  (5-minute window prevents re-scheduling after restart)
→ For each result: set status='sending', enqueue send job
→ Uses SELECT ... FOR UPDATE SKIP LOCKED (advisory lock)
  to prevent duplicate processing in scaled deployments
```

## 17.1 Logging Strategy

```
LOG FORMAT: Structured JSON (all logs)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Every log line is a JSON object with these fields:

{
  "timestamp": "2024-01-15T10:30:00.000Z",  // ISO 8601
  "level": "info",                           // debug/info/warn/error
  "service": "api",                          // api/worker/scheduler
  "request_id": "req_abc123xyz",             // unique per request
  "organization_id": "org_xyz",              // if authenticated
  "user_id": "usr_abc",                      // if authenticated
  "message": "Contact created successfully",
  "data": {                                  // structured context
    "contact_id": "con_xyz",
    "duration_ms": 47
  }
}

NO PII IN LOGS — enforced rule (see Section 14.6)

LOG LEVELS AND THEIR USAGE:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

DEBUG:
→ Only enabled in development environment
→ Never enabled in production (too verbose, costs money)
→ Used for: function entry/exit, variable inspection

INFO:
→ Normal operational events
→ Examples:
  "Campaign send job started: {campaign_id, recipient_count}"
  "User logged in: {user_id, organization_id}"
  "Contact import completed: {count, duration_ms}"
  "AI chat request processed: {tokens_used, duration_ms}"

WARN:
→ Unexpected but recoverable situations
→ Examples:
  "Rate limit approaching for organization: {org_id}"
  "OpenAI retry attempt {n} for job {job_id}"
  "SendGrid soft bounce for {email_send_id}"
  "DNS verification failed for domain: {domain}"

ERROR:
→ Failures requiring investigation
→ Examples:
  "Database query failed: {query, duration_ms, error}"
  "SendGrid API call failed: {campaign_id, error_code}"
  "Job failed after max retries: {job_id, queue, error}"
→ Always includes: error.message, error.stack

REQUEST LOGGING MIDDLEWARE:
→ Every API request logged at INFO level
→ Format: { method, path, status_code, duration_ms,
             request_id, user_id, organization_id }
→ Request body NOT logged (contains PII)
→ Response body NOT logged
→ Exception: authentication endpoints log sanitized info

LOG AGGREGATION:
→ Development: Console (pino-pretty for readability)
→ Staging/Production: stdout → collected by platform
  (Railway/Render ships logs to log aggregator)
→ Log service: Logtail (BetterStack) or Datadog Logs
→ Retention: 30 days hot, 90 days archived
→ Search: Full-text search by request_id, org_id, user_id

LIBRARY:
→ pino (Node.js) — fastest structured logger
→ pino-http for HTTP request logging middleware
→ Log transport configured via environment variable
→ Production: JSON to stdout (collected by platform)
→ Development: pino-pretty (human readable)
```

## 17.2 Error Tracking

```
TOOL: Sentry
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

SETUP:
→ Sentry DSN stored in environment variable
→ Separate Sentry projects for:
  - nexushq-api (backend errors)
  - nexushq-workers (background job errors)
  - nexushq-frontend (client-side errors)
→ Environment tags: development/staging/production

WHAT SENTRY CAPTURES:
Backend (API):
→ Unhandled exceptions in route handlers
→ Unhandled promise rejections
→ Database connection failures
→ External API failures (OpenAI, SendGrid)
→ Custom error.captureException() calls

Workers:
→ Job processing failures (automatically captured)
→ All errors in Bull job processors

Frontend:
→ JavaScript runtime errors
→ React error boundaries
→ Failed API calls (optional — use sparingly)
→ Performance issues (slow page loads)

PII SCRUBBING IN SENTRY:
→ beforeSend hook on Sentry.init():
  - Remove email fields from event context
  - Remove name fields
  - Replace with "[Filtered]"
  - Keep organization_id and user_id (non-PII IDs)
→ Default PII is disabled (sendDefaultPii: false)

SENTRY CONTEXT ENRICHMENT:
→ On every authenticated request, set Sentry scope:
  Sentry.setUser({
    id: req.user.id,
    // No email — privacy
  });
  Sentry.setTag('organization_id', req.user.organization_id);
  Sentry.setTag('user_role', req.user.role);
  Sentry.setTag('plan', req.organization.plan);

ERROR GROUPING STRATEGY:
→ Use Sentry fingerprinting to group similar errors
→ Database errors grouped by query pattern, not message
→ AI errors grouped by error type, not full message
→ Ignore: 404 errors, 401 errors, 429 errors
  (these are expected, not bugs)

ALERT ROUTING:
→ New error (first occurrence): Slack #dev-alerts
→ Error spike (> 10 in 5 minutes): Slack #dev-alerts + email
→ Critical error (unhandled in auth or billing): PagerDuty
→ Worker job failure: Slack #worker-alerts
```

## 19.1 CI/CD Pipeline

```
PLATFORM: GitHub Actions
PIPELINE STAGES:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

─────────────────────────────────────────

  Job A: Lint
    → ESLint on /src (backend)
    → ESLint on /app (frontend)
    → TypeScript type check (tsc --noEmit)
    → Fail PR if any errors

  Job B: Security Scan
    → npm audit --audit-level=high
    → Fail if high or critical vulnerabilities found
    → Snyk or Trivy for dependency scanning

Stage 2: Unit Tests (~1 min)
  → Run all unit tests (Vitest)
  → Enforce > 80% coverage on /src/services/**
  → Fail if coverage drops

Stage 3: Integration Tests (~3 min)
  → Spin up PostgreSQL + Redis (GitHub Actions services)
  → Run database migrations
  → Seed test data
  → Run all integration tests
  → Fail if any test fails

Stage 4: Build Check (~2 min)
  → Build Next.js frontend (next build)
  → Build Node.js backend (tsc or esbuild)
  → Fail if build fails

All 4 stages must pass → PR can be merged
─────────────────────────────────────────

TRIGGER: Merge to 'develop' branch
─────────────────────────────────────────

Stage 1-4: Same as above (re-run on merge)

Stage 5: Deploy to Staging (~3 min)
  → Deploy frontend to Vercel (preview environment)
  → Deploy backend to Railway (staging service)
  → Deploy workers to Railway (staging worker service)
  → Run database migrations on staging DB
  → Run smoke tests (5 critical API calls)

Stage 6: E2E Tests on Staging (~10 min)
  → Run Playwright E2E tests against staging URL
  → If pass: staging is considered validated

Notification: Slack #deployments on success/failure
─────────────────────────────────────────

TRIGGER: Merge to 'main' branch
─────────────────────────────────────────

Stage 1-6: Run everything again (fresh validation)

Stage 7: Deploy to Production
  → Require manual approval (GitHub environment protection)
  → Approver: CTO or Lead Engineer
  → Approval expires after 24 hours (must re-trigger)

  After approval:
  a. Pre-deployment snapshot:
     → Create DB backup (Neon point-in-time restore point)
     → Log deployment start to audit system

  b. Database migrations:
     → Run pending migrations
     → Verify migration success before continuing
     → If migration fails: STOP deployment, alert team

  c. Deploy workers (zero-downtime):
     → Deploy new worker version
     → Wait for in-flight jobs to complete
     → Drain old worker queues
     → New workers start processing

  d. Deploy backend (rolling deploy):
     → Railway rolling deploy (one instance at a time)
     → Health check passes before old instance removed
     → Zero downtime for users

  e. Deploy frontend:
     → Vercel deploy (atomic, instant cutover)
     → Old version still served until DNS propagates

  f. Post-deployment validation:
     → Automated smoke tests against production
     → Monitor error rate in Sentry for 10 minutes
     → If error rate spikes: trigger automatic rollback

Notification: Slack #deployments + #announcements

PIPELINE FILE STRUCTURE:
.github/workflows/
  ci.yml         → PR checks (stages 1-4)
  staging.yml    → Develop branch (stages 1-6)
  production.yml → Main branch (stages 1-7)

## 19.3 Database Migration Strategy

```
MIGRATION TOOL: Drizzle Kit (matches ORM choice)
LOCATION: /drizzle/migrations/ directory

MIGRATION RULES:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Rule 1: Migrations are always forward-only in production
  → Every migration has an up() function
  → Down migrations are written but only used in development
  → Production rollback = write a new forward migration

Rule 2: Non-destructive migrations only on existing tables
  → NEVER drop columns in production migration (mark deprecated)
  → Add columns with DEFAULT values (avoids table lock)
  → Rename columns: add new column, backfill, deprecate old,
    remove old in a separate release after code deployed
  → NEVER change column types directly (create new, migrate, drop)

Rule 3: Large table migrations use background jobs
  → If migration affects > 100,000 rows:
    Write a background migration job instead of ALTER TABLE
    Add column (nullable), deploy, backfill via job,
    add constraint in next release

Rule 4: Migration files are immutable after merge to main
  → Never edit a migration file that has been deployed
  → Always write a new migration

Rule 5: Each migration is a single logical change
  → Don't bundle unrelated schema changes in one migration
  → Makes rollback and debugging cleaner

MIGRATION PROCESS:
1. Developer runs: drizzle-kit generate:pg
   (generates migration file from schema changes)
2. Review generated SQL manually
3. Test on local DB: drizzle-kit push
4. Test on staging: migration runs automatically in CI
5. Production: migration runs as part of deployment pipeline
   before new code is deployed
   If migration fails: deployment aborted, alert sent

ZERO-DOWNTIME MIGRATION STRATEGY:
For additive changes (new columns, new tables):
  → Run migration first, deploy code second
  → Old code ignores new columns (no breakage)

For breaking changes (removing/renaming):
  Step 1: Deploy code that handles both old and new schema
  Step 2: Run migration to update schema
  Step 3: Deploy code that only uses new schema
  Step 4: Clean up compatibility code in Step 1

MIGRATION TRACKING:
  → Drizzle maintains migration history table in DB
  → Migration runs logged to deployment audit
  → Failed migrations immediately alert on-call
```

## 19.4 Rollback Strategy

```
ROLLBACK LEVELS:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

LEVEL 1: Instant Rollback (0-2 minutes)
When: Error spike detected in first 10 minutes after deploy
How:
  → Frontend (Vercel): instant rollback to previous deployment
    via Vercel dashboard or CLI: vercel rollback
  → Backend (Railway): redeploy previous Docker image
    via Railway dashboard or CLI
  → Workers: same as backend
  → Database migrations: NOT rolled back at this level
    (code reverted to handle both old and new schema)
Trigger: Automatic (if error rate > 5% for 5 minutes post-deploy)
         OR Manual (engineer decision)

LEVEL 2: Database Rollback (30-60 minutes)
When: Critical data corruption detected
How:
  → Neon point-in-time restore to pre-deployment snapshot
  → WARNING: This loses all data written since that point
  → Requires coordination to minimize data loss window
  → Only used for catastrophic migration failures
Trigger: Manual only, requires CTO approval

LEVEL 3: Feature Flag Disable (instant)
When: Bug in specific new feature (not core system)
How:
  → Toggle feature flag to disabled for all orgs or specific org
  → Feature hidden from UI immediately
  → No code deployment required
Trigger: Any engineer can do this immediately

ROLLBACK DECISION TREE:
  Error detected post-deploy
  → Is it isolated to new feature? → Disable feature flag
  → Is it widespread but DB is clean? → Level 1 rollback
  → Is there data corruption? → Level 2 (with approval)
  → Is it minor/limited impact? → Fix forward with hotfix

ROLLBACK VALIDATION:
→ After rollback: run smoke tests to confirm system healthy
→ Notify users via status page if downtime exceeded 5 minutes
→ Post-incident review within 24 hours for Level 1 or 2 rollbacks
```

```

DAILY SUMMARIES:
  Retention:    90 days
  After 90 days: Automatically deleted (daily cleanup job)

  Retention:    3 years minimum (compliance requirement)

                requests (required for security investigation)

  Revoked tokens: Retained for 7 days after revocation

  After expiry: Deleted by daily cleanup job

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

## 21.1 Full-Text Search Implementation

```
APPROACH: PostgreSQL Full-Text Search (FTS)
RATIONALE:
  → No additional infrastructure required (no Elasticsearch)
  → Sufficient for MVP scale (< 1M contacts)
  → Switch to Elasticsearch/Typesense at > 500K records
    per organization if performance degrades

FULL-TEXT SEARCH SETUP:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

CONTACTS SEARCH:

-- Add tsvector column for full-text search
ALTER TABLE contacts ADD COLUMN search_vector tsvector;

-- Function to update search vector
CREATE OR REPLACE FUNCTION contacts_search_vector_update()
RETURNS TRIGGER AS $$
BEGIN
  NEW.search_vector :=
    setweight(to_tsvector('english',
      COALESCE(NEW.first_name, '') || ' ' ||
      COALESCE(NEW.last_name, '')), 'A') ||
    setweight(to_tsvector('english',
      COALESCE(NEW.email, '')), 'A') ||
    setweight(to_tsvector('english',
      COALESCE(NEW.job_title, '')), 'B') ||
    setweight(to_tsvector('english',
      COALESCE(NEW.phone, '')), 'C');
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- Trigger to keep search_vector updated
CREATE TRIGGER contacts_search_vector_trigger
BEFORE INSERT OR UPDATE OF first_name, last_name, email,
  job_title, phone
ON contacts
FOR EACH ROW EXECUTE FUNCTION contacts_search_vector_update();

-- GIN index on search vector
CREATE INDEX idx_contacts_search
  ON contacts USING GIN(search_vector);

-- Backfill existing records
UPDATE contacts SET search_vector =
  setweight(to_tsvector('english',
    COALESCE(first_name, '') || ' ' ||
    COALESCE(last_name, '')), 'A') ||
  setweight(to_tsvector('english',
    COALESCE(email, '')), 'A') ||
  setweight(to_tsvector('english',
    COALESCE(job_title, '')), 'B');

SEARCH QUERY:
SELECT *
FROM contacts
WHERE organization_id = $org_id
  AND deleted_at IS NULL
  AND search_vector @@ plainto_tsquery('english', $query)
ORDER BY
  ts_rank(search_vector, plainto_tsquery('english', $query)) DESC,
  last_activity_at DESC NULLS LAST
LIMIT 20;

DEALS SEARCH:
-- Similar pattern for deals table
-- Fields indexed: title (A), description (B), tags (C)

COMPANIES SEARCH:
-- Fields indexed: name (A), domain (B), industry (C)

GLOBAL SEARCH (cross-entity):
-- Runs 3 queries in parallel (Promise.all):
--   contacts FTS query
--   deals FTS query
--   companies FTS query
-- Returns merged results with entity type label
-- Deduplicates by ID
-- Max 5 results per entity type in global search

## 21.3 Query Optimization Rules

```
MANDATORY QUERY PATTERNS:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

RULE 1: Always filter by organization_id first
  CORRECT:
  SELECT * FROM contacts
  WHERE organization_id = $1   ← Always first
    AND status = 'lead'
    AND deleted_at IS NULL
  ORDER BY last_activity_at DESC
  LIMIT 50;

  WRONG:
  SELECT * FROM contacts
  WHERE status = 'lead'        ← Missing org filter
  LIMIT 50;

RULE 2: Use LIMIT on all list queries
  → Maximum page size: 100 records
  → Default page size: 25 records
  → NEVER fetch all records without pagination
  → Exception: CSV exports (streaming, not in-memory load)

RULE 3: Use cursor pagination for large datasets
  → Offset pagination acceptable for < 10,000 records
  → Cursor pagination required for > 10,000 records
  → Cursor: last record's (created_at, id) tuple
  → Prevents "offset drift" as records are inserted/deleted
  → See Section 23.1 for cursor pagination standard

RULE 4: Avoid N+1 queries
  → Use JOINs or batch queries instead of per-record queries
  → Contact list: JOIN companies in same query, not per contact
  → Deal kanban: JOIN pipeline_stages to get stage details
  → Activity timeline: JOIN users for creator info in same query
  → Use DataLoader pattern for GraphQL (future)

RULE 5: Use partial indexes for common filtered queries
  → Most status queries filter WHERE status = 'open'
    AND deleted_at IS NULL
  → Partial indexes on these conditions are significantly
    smaller and faster than full table indexes

RULE 6: Use EXPLAIN ANALYZE before adding any index
  → Run EXPLAIN ANALYZE on query before and after index
  → Confirm index is actually used (not sequential scan)
  → Target: Seq Scan eliminated on tables > 10,000 rows

SLOW QUERY DETECTION:
→ PostgreSQL auto_explain extension enabled
→ log_min_duration_statement = 200ms
  (any query > 200ms logged to PostgreSQL log)
→ pg_stat_statements extension tracks query frequency
  and average duration
→ Weekly review of top 10 slowest queries
→ Index added or query rewritten if p95 > 100ms

QUERY PERFORMANCE BUDGET PER API ENDPOINT:
→ Total DB time budget per request = 50% of endpoint SLA
→ Tier 1 endpoint (200ms SLA): max 100ms in DB queries
→ Tier 2 endpoint (500ms SLA): max 250ms in DB queries
→ If DB time > budget: optimize before shipping
```

# SECTION 22: MULTI-TENANT SECURITY ENFORCEMENT

## 22.1 Enforcement Architecture

```
DEFENSE IN DEPTH — THREE LAYERS:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

LAYER 1: JWT Middleware (Request entry point)
LAYER 2: Service Layer (Business logic enforcement)
LAYER 3: PostgreSQL Row Level Security (Database safety net)

These three layers operate independently.
A bypass of Layer 1 is caught by Layer 2.
A bypass of Layer 2 is caught by Layer 3.
All three must fail simultaneously for data leakage.
This has never occurred in the tested implementation.

LAYER 1: JWT MIDDLEWARE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Implementation:
// middleware/authenticate.ts
export const authenticate = async (req, res, next) => {
  const token = req.headers.authorization?.split(' ')[1];
  if (!token) return res.status(401).json({
    error: 'UNAUTHORIZED', message: 'Authentication required'
  });

  try {
    const payload = jwt.verify(token, PUBLIC_KEY, {
      algorithms: ['RS256']
    });

    // These values come from the verified token ONLY
    // NEVER from request body, query params, or headers
    req.user = {
      id: payload.sub,
      organization_id: payload.org,  // ← This is the lock
      role: payload.role
    };

    // Verify user still exists and is active in this org
    // Cached in Redis for 5 minutes to avoid DB hit on every request
    const userValid = await verifyUserActive(
      payload.sub, payload.org
    );
    if (!userValid) return res.status(401).json({
      error: 'UNAUTHORIZED', message: 'User no longer active'
    });

    next();
  } catch (err) {
    return res.status(401).json({
      error: 'INVALID_TOKEN', message: 'Token is invalid or expired'
    });
  }
};

LAYER 2: SERVICE LAYER ENFORCEMENT
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Implementation pattern used in EVERY service function:

// services/contact.service.ts
export const getContactById = async (
  contactId: string,
  requestingUser: { id: string; organization_id: string; role: string }
) => {
  // organization_id injected from token, not from request
  const contact = await db.query(
    `SELECT * FROM contacts
     WHERE id = $1
       AND organization_id = $2    ← MANDATORY in every query
       AND deleted_at IS NULL`,
    [contactId, requestingUser.organization_id]
  );

  // Returns null if not found OR if belongs to different org
  // Caller receives 404 in both cases (no information leakage)
  if (!contact) return null;

  return contact;
};

CRITICAL RULE: organization_id is NEVER accepted from:
→ Request body (JSON payload)
→ URL query parameters
→ URL path parameters (for multi-record operations)
→ Request headers (other than Authorization)

It is ONLY sourced from: Verified JWT payload

LAYER 3: POSTGRESQL ROW LEVEL SECURITY
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Purpose: Last-resort safety net. Prevents data access
even if application code has a bug bypassing Layers 1 and 2.

Setup:
-- Create application database role (not superuser)
CREATE ROLE nexushq_app LOGIN PASSWORD '...';

-- Enable RLS on all tenant-scoped tables
ALTER TABLE contacts ENABLE ROW LEVEL SECURITY;
ALTER TABLE companies ENABLE ROW LEVEL SECURITY;
ALTER TABLE deals ENABLE ROW LEVEL SECURITY;
ALTER TABLE activities ENABLE ROW LEVEL SECURITY;
ALTER TABLE email_campaigns ENABLE ROW LEVEL SECURITY;
ALTER TABLE email_sends ENABLE ROW LEVEL SECURITY;
ALTER TABLE email_events ENABLE ROW LEVEL SECURITY;
ALTER TABLE ai_sessions ENABLE ROW LEVEL SECURITY;
ALTER TABLE ai_messages ENABLE ROW LEVEL SECURITY;
ALTER TABLE deal_insights ENABLE ROW LEVEL SECURITY;
ALTER TABLE tasks ENABLE ROW LEVEL SECURITY;
ALTER TABLE notes ENABLE ROW LEVEL SECURITY;
ALTER TABLE contact_lists ENABLE ROW LEVEL SECURITY;
ALTER TABLE pipelines ENABLE ROW LEVEL SECURITY;
ALTER TABLE pipeline_stages ENABLE ROW LEVEL SECURITY;

-- RLS Policy: app role can only see its org's data
-- The current_setting is set at the start of each DB connection
CREATE POLICY tenant_isolation ON contacts
  FOR ALL
  TO nexushq_app
  USING (
    organization_id = current_setting(
      'app.current_organization_id'
    )::uuid
  );

-- Repeat for each table above

-- Application sets this at the start of each request transaction:
-- SET LOCAL app.current_organization_id = '<org_id>';

-- This means even raw SQL queries executed by
-- application code cannot access other org's data

MIGRATION IMPACT:
→ RLS adds ~5-10ms to query execution (acceptable)
→ Admin migrations run as superuser (bypasses RLS — needed for
  schema changes that touch all orgs' data)
→ Admin DB role (for migrations): separate from app role
→ App role NEVER has superuser or BYPASSRLS privileges
```

## 22.2 Tenant Context Propagation

```
HOW ORGANIZATION CONTEXT FLOWS THROUGH THE SYSTEM:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. HTTP Request arrives
   Authorization: Bearer <JWT>

2. authenticate middleware
   → Verifies JWT
   → Extracts organization_id from payload
   → Sets req.user = { id, organization_id, role }

3. tenantContext middleware (runs after authenticate)
   → Fetches organization record (with plan, settings)
   → Attaches to request: req.organization = { id, plan, ... }
   → Sets PostgreSQL session variable:
     await db.query(
       "SET LOCAL app.current_organization_id = $1",
       [req.user.organization_id]
     );

4. Route handler calls service layer
   → Passes requestingUser object (contains org_id)
   → Service functions use org_id from this object
   → NEVER reads org_id from any other source

5. Service function runs DB query
   → WHERE organization_id = requestingUser.organization_id
   → RLS also enforces via session variable (Layer 3)

6. Result returned to route handler
   → Result never contains records from other orgs
     (enforced at both service layer and RLS)

7. Background Jobs (no HTTP context)
   → organization_id passed explicitly in job payload
   → Job processor validates org_id against DB before use
   → RLS set in job transaction:
     SET LOCAL app.current_organization_id = job.data.organization_id

8. AI Sessions
   → organization_id always in AI session record
   → Context query fetches data with org filter
   → AI response never contains cross-org data
     because input data is isolated
```

## 23.1 Pagination Standard

```
PAGINATION TYPES:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

TYPE 1: OFFSET PAGINATION
Use when: < 10,000 records, simple use cases
          User can jump to arbitrary page number

Request parameters:
  page:   integer, 1-based (default: 1)
  limit:  integer, max 100, default 25

Query:
  SELECT * FROM contacts
  WHERE organization_id = $1
  ORDER BY created_at DESC
  LIMIT $limit OFFSET (($page - 1) * $limit);

Response format:
{
  "data": [...],
  "meta": {
    "total": 1247,
    "page": 2,
    "limit": 25,
    "total_pages": 50,
    "has_next": true,
    "has_prev": true
  }
}

TYPE 2: CURSOR PAGINATION
Use when: > 10,000 records, infinite scroll,
          high-frequency inserts/deletes

Request parameters:
  cursor: string (base64-encoded cursor from previous response)
          null or absent for first page
  limit:  integer, max 100, default 25
  direction: "next" | "prev" (default: "next")

Cursor encoding:
  cursor = base64({
    created_at: "2024-01-15T10:30:00Z",
    id: "uuid-of-last-record"
  })

Query (forward pagination):
  SELECT * FROM contacts
  WHERE organization_id = $1
    AND (created_at, id) < ($cursor_created_at, $cursor_id)
    -- Uses (created_at DESC, id DESC) composite index
  ORDER BY created_at DESC, id DESC
  LIMIT $limit + 1;  -- fetch one extra to determine has_next

  If result.length > limit: has_next = true, pop last item
  If result.length <= limit: has_next = false

Response format:
{
  "data": [...],
  "meta": {
    "limit": 25,
    "has_next": true,
    "has_prev": false,
    "next_cursor": "eyJjcmVhdGVkX2F0IjoiMjAyNC0wMS0xNVQxMDozMDowMFoiLCJpZCI6InV1aWQifQ==",
    "prev_cursor": null
  }
}

WHICH ENDPOINTS USE WHICH:
→ Offset:  /contacts, /companies, /deals, /campaigns, /tasks
           (records < 100K per org in MVP, users want page jumps)
→ Cursor:  /activities (can be millions per org)
           /email_sends (can be millions per campaign)
           /email_events (can be tens of millions)
           /ai/sessions/:id/messages (long conversations)

DEFAULT SORT ORDER (if not specified by client):
→ contacts:      created_at DESC
→ companies:     created_at DESC
→ deals:         created_at DESC
→ activities:    logged_at DESC
→ campaigns:     created_at DESC
→ tasks:         due_date ASC NULLS LAST, created_at DESC
```

## 23.5 Idempotency Keys

```
PURPOSE:
Ensure that duplicate requests (due to network retries,
client bugs, or timeout-and-retry patterns) do not cause
duplicate operations, especially for non-idempotent
operations like sending emails or creating records.

ENDPOINTS REQUIRING IDEMPOTENCY KEYS:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

REQUIRED (enforced):
→ POST /campaigns/:id/send
→ POST /contacts/import
→ POST /users/invite (prevents double invitation)

OPTIONAL BUT SUPPORTED (client may provide):
→ POST /contacts
→ POST /companies
→ POST /deals
→ POST /campaigns

IMPLEMENTATION:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Client provides header on request:
  Idempotency-Key: <client-generated UUID>

Format: UUID v4
Example: Idempotency-Key: 7c2e4b1a-8f3d-4a9e-b2c1-5d8e7f0a1b2c

MISSING KEY BEHAVIOR:
→ For REQUIRED endpoints: return 400 with error
  MISSING_IDEMPOTENCY_KEY
→ For OPTIONAL endpoints: process normally, no deduplication

Server processing:
1. Receive request with Idempotency-Key header
2. Check Redis: GET idempotency:{org_id}:{key}
3. If found:
   a. Return original response (cached in Redis)
   b. Add header: Idempotency-Replay: true
   c. No operation performed (safe replay)
4. If not found:
   a. Process request normally
   b. Store result in Redis:
      SET idempotency:{org_id}:{key} <response_body>
      EXPIRE 86400  (24 hours)
   c. Return response normally

IDEMPOTENCY STORE TABLE (for critical operations):
For campaign sends, also store in DB for durability
beyond Redis TTL:

CREATE TABLE idempotency_keys (
  key             VARCHAR(100) NOT NULL,
  organization_id UUID NOT NULL,
  endpoint        VARCHAR(200) NOT NULL,

## 24.1 Feature Flag Design

```
PURPOSE:
→ Deploy code to production without releasing features
→ Enable features for specific organizations (beta testing)
→ Gradually roll out features to percentage of users
→ Instantly disable a feature if bugs are found post-launch
→ Run A/B tests on UI/UX changes

FEATURE FLAG TYPES:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

TYPE 1: Boolean Flag (on/off for entire system)
  Example: enable_ai_chat
  Use case: Emergency disable of broken feature
  Scope: System-wide (affects all organizations)

TYPE 2: Organization Flag (on/off per org)
  Example: beta_dynamic_segments
  Use case: Gradual rollout, specific org access
  Scope: Per-organization override

TYPE 3: Percentage Rollout Flag
  Example: new_kanban_ui → 20% of organizations
  Use case: Gradual A/B rollout
  Scope: Deterministic hash of org ID determines inclusion

TYPE 4: Plan Flag (linked to subscription tier)
  Example: ai_email_generator → Growth and Pro only
  Use case: Feature gating by plan
  Scope: Checked against organization.plan field
  NOTE: Plan flags are NOT in feature_flags table —
        they are hardcoded in code as plan capability checks
        and are separate from the dynamic flag system

DATABASE SCHEMA:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

CREATE TABLE feature_flags (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  key             VARCHAR(100) NOT NULL UNIQUE,
                  -- snake_case identifier: enable_ai_chat
  name            VARCHAR(255) NOT NULL,
                  -- Human readable: "AI Chat Assistant"
  description     TEXT,
  flag_type       VARCHAR(50) NOT NULL,
                  -- boolean, org_override, percentage
  default_value   BOOLEAN NOT NULL DEFAULT FALSE,
                  -- Global default: is it on or off?
  percentage      INTEGER,
                  -- 0-100, used for percentage rollout type
  enabled_for_internal BOOLEAN NOT NULL DEFAULT FALSE,
                  -- On for internal/staff organizations
  is_archived     BOOLEAN NOT NULL DEFAULT FALSE,
                  -- Archived = fully released, flag removed from code
  created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE TABLE feature_flag_overrides (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  flag_key        VARCHAR(100) NOT NULL
                  REFERENCES feature_flags(key),
  organization_id UUID NOT NULL REFERENCES organizations(id)
                  ON DELETE CASCADE,
  enabled         BOOLEAN NOT NULL,
                  -- Override value for this specific org
  override_reason TEXT,
                  -- Why this org has override (audit trail)
  created_by      VARCHAR(255),
                  -- Internal admin who set it
  created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  expires_at      TIMESTAMPTZ,
                  -- Optional expiry for temporary overrides
  PRIMARY KEY (flag_key, organization_id)
);

CREATE INDEX idx_flag_overrides_org
  ON feature_flag_overrides(organization_id);
```

## 24.3 Feature Flag Client Implementation

```
SERVER-SIDE FLAG EVALUATION:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

// lib/featureFlags.ts

interface FlagContext {
  organizationId: string;
  isInternal: boolean;
  plan: string;
}

export class FeatureFlagService {
  private cache: Map<string, boolean> = new Map();
  private CACHE_TTL_MS = 5 * 60 * 1000; // 5 minutes

  async isEnabled(flagKey: string, context: FlagContext): Promise<boolean> {
    const cacheKey = `${flagKey}:${context.organizationId}`;

    // Check Redis cache first
    const cached = await redis.get(`flag:${cacheKey}`);
    if (cached !== null) return cached === '1';

    // Evaluate flag
    const result = await this.evaluateFlag(flagKey, context);

    // Cache in Redis for 5 minutes
    await redis.setex(`flag:${cacheKey}`, 300, result ? '1' : '0');
    return result;
  }

  private async evaluateFlag(
    flagKey: string,
    context: FlagContext
  ): Promise<boolean> {
    // Step 1: Check org-level override (highest priority)
    const override = await db.query(
      `SELECT enabled FROM feature_flag_overrides
       WHERE flag_key = $1
         AND organization_id = $2
         AND (expires_at IS NULL OR expires_at > NOW())`,
      [flagKey, context.organizationId]
    );
    if (override.rows.length > 0) {
      return override.rows[0].enabled;
    }

    // Step 2: Get base flag configuration
    const flag = await db.query(
      `SELECT * FROM feature_flags
       WHERE key = $1 AND is_archived = false`,
      [flagKey]
    );
    if (!flag.rows.length) return false; // Unknown flag = disabled

    const f = flag.rows[0];

    // Step 3: Internal org check
    if (context.isInternal && f.enabled_for_internal) return true;

    // Step 4: Evaluate by flag type
    switch (f.flag_type) {
      case 'boolean':
        return f.default_value;

      case 'percentage':
        if (!f.percentage || f.percentage === 0) return false;
        if (f.percentage === 100) return true;
        // Deterministic hash: same org always gets same result
        const hash = hashOrgId(context.organizationId, flagKey);
        return (hash % 100) < f.percentage;

      case 'org_override':
        // Only enabled if explicit override exists (checked above)
        return false;

      default:
        return false;
    }
  }

  // Invalidate cache for an org (call after override changes)
  async invalidateOrg(organizationId: string): Promise<void> {
    const pattern = `flag:*:${organizationId}`;
    const keys = await redis.keys(pattern);
    if (keys.length > 0) await redis.del(...keys);
  }
}

// Singleton instance
export const featureFlags = new FeatureFlagService();

USAGE IN ROUTE HANDLERS:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

// In route handler or middleware

  organizationId: req.user.organization_id,
  isInternal: req.organization.is_internal,
  plan: req.organization.plan
});

if (!aiEnabled) {
  return res.status(403).json({
    error: 'FEATURE_NOT_AVAILABLE',
    message: 'This feature is not available for your account'
  });
}

USAGE IN MIDDLEWARE (route-level flag guard):
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

export const requireFeatureFlag = (flagKey: string) => {
  return async (req, res, next) => {
    const enabled = await featureFlags.isEnabled(flagKey, {
      organizationId: req.user.organization_id,
      isInternal: req.organization.is_internal,

      });
    }

  };
};

FRONTEND FLAG INTEGRATION:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

## 24.4 Feature Flag Rollout Process

```
STANDARD ROLLOUT SEQUENCE FOR NEW FEATURES:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

STAGE 1: CODE DEPLOYED (flag default = false)
  Who sees it:  Nobody
  Duration:     Until deployment validated (1-24 hours)
  Monitoring:   Watch error rates, nothing should change
  Rollback:     Deploy previous version (no flag needed)

STAGE 2: INTERNAL TESTING (enabled_for_internal = true)
  Who sees it:  Internal staff organizations only
  Duration:     1-7 days depending on complexity
  Monitoring:   Staff dogfooding, bug reports
  Criteria to advance:
    → No critical bugs found
    → Internal team uses it for 3+ days
    → Performance metrics within acceptable range

STAGE 3: BETA ORGANIZATIONS (org_override = enabled)
  Who sees it:  Explicitly opted-in beta organizations
                (typically 5-20 early adopter customers)
  Duration:     7-14 days
  How to add:   INSERT into feature_flag_overrides
  Monitoring:   Customer feedback, error rates,
                performance metrics per beta org
  Criteria to advance:
    → No critical bugs
    → Beta customers actively using feature
    → Positive feedback, no major UX issues
    → Support tickets related to feature are manageable

STAGE 4: PERCENTAGE ROLLOUT (flag_type = percentage)
  Start:        percentage = 10 (10% of organizations)
  Duration:     3-5 days per increment
  Increments:   10% → 25% → 50% → 100%
  Monitoring:   Compare metrics between enabled/disabled orgs
    → Error rates by flag cohort
    → Feature usage rate in enabled orgs
    → Performance impact (latency, DB queries)
    → Support ticket volume
  Rollback:     Reduce percentage to 0 immediately

STAGE 5: FULL RELEASE (default_value = true or 100%)
  Who sees it:  All organizations
  Action:       Update default_value = true in database
                (or percentage = 100)
  Announcement: Release notes, in-app notification if major

STAGE 6: FLAG RETIREMENT (is_archived = true)
  When:         Feature stable for 30+ days at 100%
  Action:
    1. Remove all flag checks from code
    2. Set is_archived = true in database
    3. Delete feature_flag_overrides for this key
    4. Deploy (feature now always-on in code)
  Purpose:      Reduce technical debt from flag proliferation

EMERGENCY DISABLE PROCEDURE:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

When a feature has a critical bug post-release:
1. Any engineer runs:
   UPDATE feature_flags
   SET default_value = false
   WHERE key = 'problematic_feature_key';

2. Flush Redis flag cache:
   redis-cli FLUSHDB  -- or specific key pattern flush

3. Feature disabled system-wide within < 1 minute
   (cache TTL means up to 5 min for full propagation
    — acceptable for emergency)

4. Investigate and fix the bug

5. When fixed, re-enable via percentage rollout
   (don't jump straight back to 100%)

ADMIN UI FOR FLAG MANAGEMENT:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Internal admin panel at /internal/flags
(separate from customer-facing /settings)

Features:
→ List all feature flags with current state
→ Toggle default value
→ Add/remove organization overrides
→ Adjust rollout percentage
→ Archive retired flags
→ View which orgs have overrides
→ Audit log of all flag changes

Access: Internal staff only
        Not accessible to customers
        Separate authentication from main app
        (IP allowlist to office/VPN IP ranges)
```

CREATE TABLE notifications (

CREATE INDEX idx_notifications_user_unread
  ON notifications(user_id, created_at DESC)
  WHERE is_read = FALSE;

  ON notifications(user_id, created_at DESC);

## F.1 Error Classification

```
ERROR TAXONOMY:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

CLASS 1: OPERATIONAL ERRORS (expected, handled gracefully)
  Definition:  Errors that occur in normal operation
               The system knows how to handle them
               They are not bugs

  Examples:
  → Duplicate email on contact create
  → Invalid JWT token
  → Missing required field
  → Rate limit exceeded
  → Campaign already sent

  Handling:
  → Return appropriate HTTP status code
  → Return structured error response
  → Do NOT log to Sentry
  → Do NOT trigger alerts
  → Log at WARN level if unexpected frequency

CLASS 2: PROGRAMMER ERRORS (bugs, need fixing)
  Definition:  Errors that should not happen
               Indicate a bug in the code

  Examples:
  → Unhandled promise rejection
  → TypeError: Cannot read property of undefined
  → Missing required function argument
  → Database constraint violation (unexpected)

  Handling:
  → Return HTTP 500
  → Log full stack trace at ERROR level
  → Send to Sentry immediately
  → Trigger alert if frequency spikes

CLASS 3: EXTERNAL SERVICE ERRORS (degraded, retry)
  Definition:  Third-party API failures beyond our control

  Examples:
  → OpenAI API returns 503
  → SendGrid API times out
  → Database connection pool exhausted temporarily
  → Redis connection refused

  Handling:
  → Retry with exponential backoff (see Section 16.3)
  → Return 503 to client if retries exhausted
  → Log at WARN level on first failure
  → Log at ERROR level after max retries
  → Alert if sustained failure > 5 minutes

CLASS 4: SECURITY ERRORS (log, alert, investigate)
  Definition:  Requests that indicate malicious activity

  Examples:
  → Cross-tenant access attempt
  → JWT with tampered claims
  → Brute force login
  → SQL injection attempt (blocked by parameterized queries)
  → Prompt injection in AI chat

  Handling:
  → Return 401/403/400 (do not reveal reason)
  → Log at ERROR level with full context
  → Create audit log entry (severity: critical/warning)
  → Trigger security alert
  → Do NOT send to Sentry (security event, not a bug)
```

## F.2 Global Error Handler

```typescript
// middleware/errorHandler.ts

import { Request, Response, NextFunction } from 'express';
import * as Sentry from '@sentry/node';
import { logger } from '../lib/logger';

export class AppError extends Error {
  constructor(
    public statusCode: number,
    public errorCode: string,
    public message: string,
    public details?: object[]
  ) {
    super(message);
    this.name = 'AppError';
  }
}

export class ValidationError extends AppError {
  constructor(details: Array<{field: string; message: string;
                               code: string}>) {
    super(422, 'VALIDATION_ERROR',
          'Request validation failed', details);
    this.name = 'ValidationError';
  }
}

export class NotFoundError extends AppError {
  constructor(resource = 'Resource') {
    super(404, 'NOT_FOUND', `${resource} not found`);
    this.name = 'NotFoundError';
  }
}

export class ForbiddenError extends AppError {
  constructor(message = 'Insufficient permissions') {
    super(403, 'INSUFFICIENT_PERMISSIONS', message);
    this.name = 'ForbiddenError';
  }
}

export class ConflictError extends AppError {
  constructor(errorCode: string, message: string) {
    super(409, errorCode, message);
    this.name = 'ConflictError';
  }
}

// Global error handler middleware
// Must be last middleware registered in Express
export const globalErrorHandler = (
  err: Error,
  req: Request,
  res: Response,
  next: NextFunction
) => {
  const requestId = req.headers['x-request-id'] as string
    || generateRequestId();

  // CASE 1: Known operational error
  if (err instanceof AppError) {
    logger.warn({
      message: 'Operational error',
      error_code: err.errorCode,
      status_code: err.statusCode,
      request_id: requestId,
      path: req.path,
      method: req.method,
      organization_id: req.user?.organization_id,
    });

    return res.status(err.statusCode).json({
      success: false,
      error: err.errorCode,
      message: err.message,
      request_id: requestId,
      ...(err.details && { details: err.details }),
    });
  }

  // CASE 2: Database unique violation (handle as conflict)
  if (err.message?.includes('unique constraint') ||
      (err as any).code === '23505') {
    return res.status(409).json({
      success: false,
      error: 'CONFLICT',
      message: 'A duplicate record already exists',
      request_id: requestId,
    });
  }

  // CASE 3: Unknown error (programmer error / bug)
  logger.error({
    message: 'Unhandled error',
    error: err.message,
    stack: err.stack,
    request_id: requestId,
    path: req.path,
    method: req.method,
    organization_id: req.user?.organization_id,
  });

  // Send to Sentry with full context
  Sentry.withScope((scope) => {
    scope.setTag('request_id', requestId);
    scope.setTag('path', req.path);
    if (req.user) {
      scope.setUser({ id: req.user.id });
      scope.setTag('organization_id', req.user.organization_id);
    }
    Sentry.captureException(err);
  });

  // Never expose internal error details to clients
  return res.status(500).json({
    success: false,
    error: 'INTERNAL_ERROR',
    message: 'An unexpected error occurred. ' +
             'Our team has been notified.',
    request_id: requestId,
  });
};
```

async function checkContactLimit(orgId: string) {
  const org = await getOrg(orgId);
  const limit = PLAN_LIMITS[org.plan].contacts;
  if (limit === -1) return; // Unlimited

  const count = await db.query(
    `SELECT COUNT(*) FROM contacts
     WHERE organization_id = $1
       AND deleted_at IS NULL`,
    [orgId]
  );

  if (parseInt(count.rows[0].count) >= limit) {
    throw new AppError(403, 'PLAN_LIMIT_EXCEEDED',
      `Your ${org.plan} plan allows ${limit} contacts. ` +
      `Upgrade to add more contacts.`
    );
  }
}

## L.3 Common Runbooks

```
RUNBOOK 1: API Server Not Responding
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Symptoms: Health check failing, 5xx errors

Steps:
1. Check Railway/Render dashboard for deploy status
2. Check API server logs for startup errors
3. Verify DATABASE_URL is set and DB is reachable
   psql $DATABASE_URL -c "SELECT 1;"
4. Verify REDIS_URL is set and Redis is reachable
   redis-cli -u $REDIS_URL ping
5. Check for OOM (Out of Memory) kills in server logs
6. Restart service via Railway/Render dashboard
7. If restart fails: check for failed migrations
   drizzle-kit push --dry-run
8. If migration issue: revert via Railway deployment history

RUNBOOK 2: Campaign Not Sending
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Symptoms: Campaign status stuck at 'sending', no progress

Steps:
1. Check Bull dashboard: are workers running?
   URL: /admin/queues (admin login required)
2. Check email:campaigns queue depth and processing rate
3. If queue depth growing but not processing:
   → Worker process may be down
   → Restart worker service on Railway/Render
4. Check for failed jobs in Bull dashboard
   → Click failed jobs, view error message
   → Common: SendGrid API key invalid
     Verify: SENDGRID_API_KEY environment variable
5. Check SendGrid status: status.sendgrid.com
   If SendGrid down: jobs will retry automatically
6. If worker running but jobs failing:
   → Check worker logs for error details
   → Common: Database connection error
     → Check DATABASE_URL, restart worker

RUNBOOK 3: OpenAI Errors / AI Not Responding
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Symptoms: AI chat returns errors, email generation fails

Steps:
1. Check OpenAI status: status.openai.com
2. If OpenAI has active incident:
   → AI features will auto-fail gracefully
   → No action needed, wait for OpenAI recovery
   → Consider posting status update on status page
3. If OpenAI operational but still failing:
   → Verify OPENAI_API_KEY is valid
     curl -H "Authorization: Bearer $OPENAI_API_KEY" \
       https://api.openai.com/v1/models
   → Check if monthly spend limit reached in OpenAI dashboard
   → Check if rate limits exceeded (too many requests)
4. If rate limited: reduce ai:insights job concurrency
   WORKER_CONCURRENCY_AI=1 (restart worker)
5. Check Redis ai_quota keys for user limits:
   redis-cli GET ai_quota:{user_id}:{YYYY-MM}

RUNBOOK 4: Database Performance Degradation
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Symptoms: Slow API responses, Tier 1 endpoints > 500ms

Steps:
1. Connect to database and check active queries:
   SELECT pid, now() - pg_stat_activity.query_start AS duration,
          query, state
   FROM pg_stat_activity
   WHERE (now() - pg_stat_activity.query_start) > interval '5 seconds'
   ORDER BY duration DESC;

2. Kill long-running queries (if blocking):
   SELECT pg_terminate_backend({pid});

3. Check for table bloat or missing indexes:
   Run EXPLAIN ANALYZE on the slow endpoint's primary query
   Look for: Sequential Scan on large tables

4. Check connection pool:
   SELECT count(*) FROM pg_stat_activity
   WHERE datname = 'nexushq_prod';
   If > 80: PgBouncer may need scaling

5. Check for lock contention:
   SELECT * FROM pg_locks l
   JOIN pg_stat_activity a ON l.pid = a.pid
   WHERE NOT granted;

6. If specific query causing issues:
   → Add LIMIT (if missing)
   → Add missing index
   → Schedule for off-peak maintenance

RUNBOOK 5: Security Alert — Cross-Tenant Access
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Symptoms: Audit log shows suspicious_access_pattern (SEV1)

Steps:
1. IMMEDIATELY: Identify the user and organization
   SELECT * FROM audit_logs
   WHERE event_type = 'suspicious_access_pattern'
   ORDER BY occurred_at DESC LIMIT 10;

2. IMMEDIATELY: Revoke all sessions for affected user
   UPDATE refresh_tokens
   SET revoked_at = NOW()
   WHERE user_id = '{affected_user_id}';
   -- Flush Redis sessions:
   redis-cli DEL "session:{user_id}:*"

3. Assess: Was actual data accessed or just attempted?
   Check audit_log metadata field for details

4. If data was accessed: this is a data breach
   → Notify CTO immediately
   → Begin breach assessment
   → Consult legal/compliance team
   → GDPR 72-hour notification may be required

5. If only attempted (blocked by security layers):
   → Document in incident log
   → Investigate how the attempt was made
   → Check if it's a bug or malicious action
   → Suspend organization if malicious confirmed

6. Root cause analysis:
   → Was RLS working? (check PostgreSQL session settings)
   → Was service layer check working?
   → Was JWT validation working?
   → Identify which layer failed or was bypassed

```
TECHNICAL TERMS USED IN THIS DOCUMENT:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

TERM                  DEFINITION
──────────────────────────────────────────────────────────────
Activity Timeline     Chronological log of all events
                      on a contact, deal, or company.
                      Combination of system-generated
                      and manually logged entries.

Bull / BullMQ         Redis-backed job queue library
                      for Node.js. Handles background
                      job processing, retries, and
                      scheduling.

Cursor Pagination     Pagination using a pointer (cursor)
                      to the last seen record rather than
                      a page number. More stable for
                      large or frequently-changing datasets.

Dead Letter Queue     Storage for jobs that have failed
                      after all retry attempts. Requires
                      manual intervention to resolve.

DKIM                  DomainKeys Identified Mail.
                      A digital signature added to emails
                      proving they were not tampered with
                      during delivery.

DMARC                 Domain-based Message Authentication,
                      Reporting & Conformance. Policy that
                      tells email providers what to do when
                      SPF or DKIM checks fail.

Feature Flag          A configuration switch that enables
                      or disables a feature without
                      deploying code. Allows safe,
                      gradual feature releases.

GIN Index             Generalized Inverted Index.
                      PostgreSQL index type optimized
                      for array columns, JSONB, and
                      full-text search vectors.

Idempotency Key       A client-provided unique token
                      that ensures a request can be
                      safely retried without producing
                      duplicate effects.

Intent Classification Analyzing a user's natural language
                      input to determine what type of
                      data or action they need. Used
                      in the AI assistant to decide
                      what CRM data to fetch.

Multi-tenant          A software architecture where a
                      single application serves multiple
                      customers (tenants) with complete
                      data isolation between them.

Organization          The top-level entity in NexusHQ.
                      One organization = one customer
                      (company) using the platform.
                      All data belongs to an organization.

p50 / p95 / p99       Percentile measurements for
                      response times. p95 = 95% of
                      requests complete within this time.
                      p50 = median response time.

Partial Index         A PostgreSQL index that only
                      includes rows matching a WHERE
                      condition. Smaller and faster
                      than a full index.

PII                   Personally Identifiable Information.
                      Data that can identify a specific
                      individual: names, emails,
                      phone numbers.

RLS                   Row Level Security. PostgreSQL
                      feature that restricts which rows
                      a database role can access.
                      Used as a safety net for
                      multi-tenant isolation.

RS256                 RSA Signature with SHA-256.
                      Asymmetric algorithm used for
                      JWT signing. Private key signs,
                      public key verifies.

Sender Domain         The email domain used to send
                      campaigns. Must be verified via
                      SPF, DKIM, and DMARC records.

SPF                   Sender Policy Framework. A DNS
                      record that lists which servers
                      are authorized to send email
                      on behalf of a domain.

tsvector              PostgreSQL data type for full-text
                      search. Stores pre-processed
                      lexemes for efficient text search.

Vector Database       Database optimized for storing and
                      searching high-dimensional vectors
                      (embeddings). Used for semantic/
                      AI-powered search (Phase 3).

Webhook               HTTP callback sent by an external
                      service (SendGrid, Stripe) when
                      an event occurs. NexusHQ receives
                      these to process email events
                      and billing updates.
